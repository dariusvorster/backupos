# What is BackupOS

BackupOS is a unified backup management platform for homelabs and small businesses. It wraps [Restic](https://restic.net) in a structured web UI with scheduled jobs, cross-platform agents, restore verification, cost forecasting, and disaster recovery tooling.

## What it replaces

| Without BackupOS | With BackupOS |
|---|---|
| Proxmox Backup Server alone | BackupOS + PBS as a storage backend |
| Manual Restic cron scripts | Scheduled jobs with pre-flight checks |
| Paid SaaS backup tools (Veeam, Acronis) | Self-hosted, data stays yours |
| Scattered shell scripts per machine | One UI for every source and agent |

## What it adds

- **Verification** — scheduled restore tests that prove backups actually work
- **Cost forecasting** — storage growth tracking and projected spend per repository
- **DR Mode** — guided recovery interface with restore specs for whole-host incidents
- **Cross-platform agents** — single binary for Linux and Windows
- **Tamper-evident audit log** — SHA-256 hash-chained event log for compliance
- **Operational logs** — structured per-job and per-agent log stream with filtering

## Architecture at a glance

```
┌─────────────┐     ┌──────────────────────────────────┐
│  BackupOS   │────▶│  Agents (Linux / Windows)        │
│  Web UI     │     │  ├── Filesystem sources           │
└─────────────┘     │  ├── Docker / database sources    │
                    │  └── Proxmox VM / LXC sources     │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │  Restic repositories              │
                    │  (R2 · B2 · S3 · local NAS)      │
                    └───────────────────────────────────┘
```

The web app runs as a Docker container on any Linux host. Agents run on each machine that has sources to back up. Repositories are standard Restic-compatible object-storage buckets — no proprietary format.

# Why BackupOS

## The three problems

### 1. Trust — does my backup actually work?

Most backup tools report success based on the job completing without error. They don't prove the data is restorable. BackupOS schedules automated restore verification: it actually restores a snapshot to a sandbox environment and checks the result. Without this, a green checkbox is a guess.

### 2. Cost — what does this storage cost me?

Object storage is inexpensive but not free, and usage grows quietly. BackupOS tracks per-repository storage usage, forecasts 30/90/365-day growth trends, and alerts when usage approaches configurable thresholds. You see the bill before it arrives.

### 3. Recovery — how do I actually restore when it matters?

Restore procedures are almost always undocumented, or documented in a wiki nobody updated in two years. BackupOS Restore Specs are declarative YAML files that live alongside your job config. DR Mode presents them as a guided step-by-step checklist during an incident, when calm thinking is hardest.

## Feature comparison

| Feature | BackupOS | PBS | Raw Restic | Veeam |
|---|---|---|---|---|
| Web UI | ✅ | ✅ | ❌ | ✅ |
| Linux sources | ✅ | ✅ | ✅ | ✅ |
| Windows sources | ✅ | ❌ | ✅ | ✅ |
| Database-aware backup | ✅ | ❌ | ❌ | ✅ |
| Restore verification | ✅ | ❌ | ❌ | ✅ |
| Cost forecasting | ✅ | ❌ | ❌ | ❌ |
| DR Mode / restore specs | ✅ | ❌ | ❌ | partial |
| Self-hosted | ✅ | ✅ | ✅ | ✅ |
| Open source | ✅ | ✅ | ✅ | ❌ |
| Price | Free | Free | Free | $$$ |

## When to use BackupOS

- You manage three or more machines that all need backups
- You want proof that backups work, not just a green status badge
- You need an audit trail for compliance or incident post-mortems
- You want one UI for Linux and Windows sources

## When not to use BackupOS

- **Single machine, simple filesystem** — raw Restic + cron is simpler and equally effective
- **Proxmox-only, VM backup only** — PBS alone handles this well and is already integrated
- **Enterprise SAP / tape library** — you need Veeam or a purpose-built enterprise tool

# Architecture overview

## Components

### Web app

The central control plane. Runs as a Docker container (image: `backupos/web`). Stores all configuration and state in a local SQLite database. Exposes the browser UI and a REST API consumed by agents.

**Responsibilities:**
- Job scheduling and orchestration
- Agent enrolment and health monitoring
- Snapshot browsing and tagging
- Restore spec management and DR Mode
- Audit and operational logging
- Alert rules and notifications

### Agents

Lightweight processes that run on source machines. A single binary (`ios-agent`) supports both Linux and Windows, shared with the Infra OS product family for consistency.

**Responsibilities:**
- Execute backup jobs as instructed by the web app
- Stream structured logs back to the web app
- Report capabilities (available source types, disk space)
- Run pre-flight checks on demand

Agents communicate with the web app over HTTPS. No inbound ports are required on agent machines.

### Repositories

Standard Restic repositories stored in object storage or on local filesystems. BackupOS does not introduce a proprietary format — any Restic-compatible tool can read the data.

**Supported backends:**
- Cloudflare R2 (recommended for zero egress cost)
- Backblaze B2
- AWS S3 and S3-compatible (MinIO, Wasabi)
- Local filesystem or NAS (NFS, SMB)

### Hypervisor integration

For Proxmox, BackupOS talks directly to the Proxmox API to trigger VM and LXC snapshots without requiring an in-guest agent. This covers the full VM even if the guest OS is inaccessible.

## Data flow — a typical backup run

```
1. Scheduler triggers job at the configured cron time
2. Web app sends run instruction to the assigned agent
3. Agent runs pre-flight checks (disk space, source reachable, repo accessible)
4. Agent executes: pre-hook → restic backup → post-hook
5. Agent streams structured log lines to the web app in real time
6. On completion, agent reports snapshot ID, bytes transferred, duration
7. Web app stores run record, updates health score, fires alerts if configured
8. (Optional) Scheduled verification test restores a file from the new snapshot
```

## Persistence

The web app uses a single SQLite file (`backupos.db`) for all relational data. This makes backup of the control plane trivial — copy the file. For high-availability deployments, a read replica or periodic export to S3 is recommended.

Restic repositories are the source of truth for backup data. The BackupOS database is an index and control plane — losing it does not lose your backups.
```

# Terminology

| Term | Definition |
|---|---|
| **Agent** | The process running on a source machine that executes backup jobs. A single binary supports Linux and Windows. |
| **Repository** | A Restic-compatible storage location (R2 bucket, B2 bucket, local path). Encrypted at rest. |
| **Job** | A configured backup task: one source, one repository, one schedule, optional hooks. |
| **Run** | A single execution of a job. Has phases: pre-flight → pre-hook → backup → post-hook → (optional) verification. |
| **Source** | What gets backed up. Types: filesystem path, Docker volume, Docker container, PostgreSQL, MySQL, Redis, SQLite, Proxmox VM, Proxmox LXC, Windows system (VSS). |
| **Snapshot** | An immutable point-in-time copy of the source, created by Restic. Deduplicated and encrypted. |
| **Tag** | A string label attached to a snapshot (e.g. `pre-migration`, `v2.3.0`). Used for filtering and retention policy overrides. |
| **Pin** | Marks a snapshot as protected. Pinned snapshots are excluded from forget/prune runs. |
| **Hold** | Like a pin but temporary. Used during migrations: set a hold, do the work, release the hold. |
| **Pre-flight** | A set of checks run before a backup: disk space, source reachable, repository accessible, last run status. |
| **Hook** | A script or HTTP call executed before or after a backup run. Common uses: quiesce a database before backup, send a webhook after completion. |
| **Restore spec** | A declarative YAML file describing how to recover a system or service step-by-step. Used in DR Mode. |
| **DR Mode** | A guided recovery interface. Activated manually or by a critical alert. Presents restore specs as a checklist. |
| **Verification** | A scheduled test that actually restores data from a snapshot and validates the result. Proves the backup is usable. |
| **Monitor** | Read-only observation of a third-party backup system (PBS, Borg, Restic scripts) without managing it. |
| **Health score** | A 0–100 score (A–F letter grade) per agent or repository, computed from recent run success rate, verification results, and snapshot freshness. |
| **Escrow** | Optional encrypted backup of a repository's encryption password, recoverable via a master passphrase. Prevents permanent data loss if the password is forgotten. |
| **Audit log** | A tamper-evident log of all configuration changes and security events, linked by SHA-256 hash chain. |
| **Operational log** | Structured, per-component log stream for debugging and observability. Separate from the audit log. |
| **API token** | A long-lived credential for programmatic access to the BackupOS API. Scoped to read-only or read-write. |
```

# Install BackupOS (self-hosted)

## Prerequisites

- Docker 24+ and Docker Compose v2
- 2 GB RAM (4 GB recommended)
- 10 GB disk for the application (backup data lives in your repositories, not here)
- A Linux host (x86_64 or arm64)

## 1. Create a project directory

```bash
mkdir backupos && cd backupos
```

## 2. Create docker-compose.yml

```yaml
services:
  web:
    image: backupos/web:latest
    restart: unless-stopped
    ports:
      - "3000:3000"
    volumes:
      - backupos-data:/data
    environment:
      - DATABASE_URL=file:/data/backupos.db
      - ENCRYPTION_KEY=change-this-to-a-random-64-char-hex-string
      - NEXTAUTH_SECRET=change-this-to-a-random-32-char-string
      - NEXTAUTH_URL=http://localhost:3000

volumes:
  backupos-data:
```

**Important:** Replace `ENCRYPTION_KEY` and `NEXTAUTH_SECRET` with random values:

```bash
openssl rand -hex 32   # NEXTAUTH_SECRET
openssl rand -hex 32   # ENCRYPTION_KEY (use 64 hex chars = 32 bytes)
```

## 3. Start the container

```bash
docker compose up -d
```

## 4. Open the setup wizard

Navigate to `http://localhost:3000`. On first boot you'll see the setup wizard:

1. **Create admin account** — enter your email and a strong password
2. **Instance name** — a friendly name shown in the UI (e.g. "Homelab")
3. **Enable TOTP** — strongly recommended; scan the QR code with your authenticator app

## Verify

```bash
docker compose ps
# Should show: web   running
```

Check the logs if the container is unhealthy:

```bash
docker compose logs -f web
```

## Troubleshooting

**Port 3000 already in use:** Change the left side of the port mapping to any free port, e.g. `"8080:3000"`.

**ENCRYPTION_KEY error on startup:** The key must be exactly 64 hex characters (32 bytes). Use `openssl rand -hex 32` to generate one.

**Database permission error:** The `/data` volume must be writable. On some hosts you may need to `chown -R 1000:1000 backupos-data`.
```

## 2. Connect your first agent

Cloud-hosted BackupOS uses a relay connection so agents don't need inbound firewall rules. When you add an agent, you get a one-line install command:

```bash
curl -fsSL https://backupos.app/install | sh -s -- --token <your-enrolment-token>

The agent connects outbound to the relay. No port forwarding required.

Backup data never passes through BackupOS servers. Agents write directly to your configured repository (R2, B2, S3). BackupOS only sees metadata: job status, snapshot IDs, byte counts.

# Enrol your first agent

The BackupOS agent is a single binary (`ios-agent`) that runs as a system service and executes backup jobs on behalf of the web app.

## Linux

### 1. Get the enrolment token

In the BackupOS UI, go to **Agents → Enrol agent**. Copy the enrolment token shown.

### 2. Run the one-line installer

```bash
curl -fsSL https://your-backupos-host:3000/install/agent | \
  sudo sh -s -- --token <enrolment-token>
```

The installer:
1. Downloads the `ios-agent` binary to `/usr/local/bin/ios-agent`
2. Creates a systemd service `ios-agent.service`
3. Starts the service and connects to the web app

### 3. Verify in the UI

Refresh the **Agents** page. Your new agent should appear within 30 seconds with status **Connected**.

### 4. Check capabilities

Click the agent row to see its detected capabilities:
- Available source types (filesystem, Docker, Proxmox, etc.)
- Free disk space on key mount points
- OS and architecture

## Windows

### 1. Download the installer

From the BackupOS UI: **Agents → Enrol agent → Windows**. Download `ios-agent-setup.exe`.

### 2. Run the installer

Double-click `ios-agent-setup.exe`. Enter the enrolment token when prompted. The installer registers a Windows Service named **BackupOS Agent**.

### 3. Verify

Check **Services** (services.msc) — `BackupOS Agent` should be Running. Then verify in the UI.

## Troubleshooting

**Agent appears but shows Disconnected:** Check that the agent machine can reach the BackupOS host on port 3000 (or your configured port).

**curl: SSL certificate error:** If you're using a self-signed cert, add `--insecure` to the curl command. In production, use a valid certificate.

# Connect a repository

A repository is where backup data is stored. BackupOS uses [Restic](https://restic.net) as its storage engine — all repositories are standard Restic repositories.

## Recommended: Cloudflare R2

Cloudflare R2 has zero egress fees, making it the cheapest option for most homelab setups. You pay only for storage (currently $0.015/GB/month).

### 1. Create an R2 bucket

1. Log into [dash.cloudflare.com](https://dash.cloudflare.com)
2. Go to **R2 → Create bucket**
3. Name it (e.g. `homelab-backups`), choose a region
4. Go to **R2 → Manage R2 API tokens → Create API token**
5. Scope: **Object Read & Write** on your bucket
6. Note the **Access Key ID** and **Secret Access Key**

### 2. Add the repository in BackupOS

Go to **Repositories → Add repository**:

| Field | Value |
|---|---|
| Name | anything (e.g. "R2 homelab") |
| Backend | S3 / R2 |
| Endpoint | `https://<account-id>.r2.cloudflarestorage.com` |
| Bucket | your bucket name |
| Access key ID | from step 1 |
| Secret access key | from step 1 |
| Encryption password | **choose a strong, unique password** |

### 3. Enable escrow (strongly recommended)

If you forget the encryption password, the backup data is permanently inaccessible. Enable escrow: BackupOS encrypts the password with your master passphrase and stores it. You can recover it later via **Settings → Recover repository password**.

Click **Initialise repository**. BackupOS calls `restic init` and stores the connection details.

## Other backends

| Backend | Endpoint format |
|---|---|
| Backblaze B2 | `b2:<bucket-name>` |
| AWS S3 | `s3:s3.amazonaws.com/<bucket>` |
| MinIO (local) | `s3:http://minio:9000/<bucket>` |
| Local path | `/mnt/nas/backups` |

## Verify

After adding: the repository card shows **Accessible** with green status. If it shows **Error**, check the credentials and endpoint URL.

# Run your first backup

## 1. Open the new job wizard

Go to **Jobs → New job**.

## 2. Configure the job

**Step 1 — Source:**
- Agent: select the agent you enrolled
- Source type: **Filesystem**
- Path: `/home` (a safe first target — not too large, easy to verify)

**Step 2 — Repository:**
- Select the repository you connected

**Step 3 — Schedule:**
- Keep the default (daily at 02:00). You can change this later.
- Enable **Auto pre-flight** — runs checks 15 minutes before each scheduled run

**Step 4 — Review:**
- Give the job a name (e.g. "Home directory — server-01")
- Click **Create job**

## 3. Run it manually

On the job detail page, click **Run now**. The job moves to **Running** status.

Watch the log stream update in real time on the **Jobs → [job name] → Runs → [run]** page. You'll see Restic output as it scans and uploads data.

## 4. See the snapshot

When the run completes (status: **Success**), go to **Snapshots**. Filter by your repository. You should see a new snapshot with today's date and your source path.

## What "success" means

A successful run means:
- Restic completed without error
- All source files were scanned
- New/changed files were uploaded to the repository
- The snapshot ID was recorded in BackupOS

It does **not** mean the backup is restorable — that's what verification is for.

# Run your first restore

Do this every time you onboard a new source. Knowing a backup exists is not the same as knowing it's usable.

## 1. Find the snapshot

Go to **Snapshots**. Click the snapshot created by your first backup run.

## 2. Start a file restore

Click **Restore → Restore file**. The restore wizard opens.

**Step 1 — Select file:**
- Browse the snapshot's file tree
- Pick any file you recognise (e.g. `/home/yourname/.bashrc`)

**Step 2 — Destination:**
- Target path: `/tmp/restore-test` (a safe temporary directory)
- Agent: the agent that has access to the destination

**Step 3 — Confirm:**
- Review the details and click **Start restore**

## 3. Verify the restored file

The restore runs as a job. When it completes:

```bash
# On the target machine
cat /tmp/restore-test/.bashrc
# Should show your actual .bashrc content
```

If the file is there and correct: your backup is confirmed restorable.

## 4. Clean up

```bash
rm -rf /tmp/restore-test
```

## Why this matters

> "Untested backups are not backups. They are hopes."

Run a restore test at least once per source when you set it up. Then set up **Verification** to automate this — BackupOS can schedule restore tests on a cadence and alert you if they fail.

# Jobs

A job combines a **source** (what to back up), a **repository** (where to store it), a **schedule** (when to run), and optional **hooks** (what to do before and after).

## Lifecycle

```
idle → queued → running → success
                        ↘ failed
                        ↘ cancelled
```

A job is **idle** between scheduled runs. The scheduler **queues** it at the configured cron time. The assigned agent picks it up and moves it to **running**. On completion: **success**, **failed**, or **cancelled** (if manually stopped).

## Job dependencies

Jobs can depend on other jobs. Set a dependency to ensure Job B only runs after Job A succeeds. Useful for: back up the database, then back up the application data that references it.

## Templates

Recurring job patterns (e.g. "nightly PostgreSQL backup") can be saved as templates and reused when adding new jobs of the same type.
```

# Runs

A run is one execution of a job. Each run has a unique ID and proceeds through these phases:

| Phase | Description |
|---|---|
| Pre-flight | Checks: disk space, source reachable, repo accessible |
| Pre-hook | Optional script or HTTP call before the backup |
| Backup | Restic backup execution |
| Post-hook | Optional script or HTTP call after the backup |
| Verification | Optional: restore a file to confirm the snapshot is usable |

The **Runs** page for a job shows the full history. Click a run to see the real-time log stream, phase timeline, bytes transferred, duration, and the Restic snapshot ID.

# Repositories

A repository is a [Restic](https://restic.net)-compatible storage location. All data is encrypted client-side before leaving the agent — the storage backend never sees plaintext.

## Supported backends

Cloudflare R2, Backblaze B2, AWS S3, S3-compatible (MinIO, Wasabi, IDrive), and local filesystem paths.

## Encryption

Each repository has its own encryption password set when the repository is initialised. This password is never stored in the BackupOS database — only in memory during use and optionally in escrow.

## Key escrow

Enable escrow to let BackupOS store an encrypted copy of the repository password, recoverable with your master passphrase. Without escrow, losing the password means permanent data loss.

title: Snapshots
description: Immutable, deduplicated, encrypted point-in-time copies of your source data.

# Snapshots

A snapshot is an immutable point-in-time copy created by Restic at the end of a successful backup run. Restic's content-addressed storage means unchanged data is never uploaded twice — only the delta is stored.

## Tags, pins, and holds

- **Tag:** a string label (e.g. `pre-migration`, `v1.2.3`). Used for filtering and retention overrides.
- **Pin:** protects a snapshot from forget/prune permanently.
- **Hold:** temporary protection during migrations or risky changes. Set it, do the work, release it.

## Retention policy

Restic's `forget` command removes old snapshots according to a policy (keep last N, keep daily for 30 days, etc.). BackupOS applies the policy per-repository on a configurable schedule.

## Diff view

Compare two snapshots to see which files changed between runs.

# Agents

The BackupOS agent (`ios-agent`) is a single binary that runs as a system service on Linux and Windows machines. It executes jobs on behalf of the web app, streams logs back in real time, and reports health status.

## Capabilities

An agent reports its available capabilities on enrolment: source types it can serve (filesystem, Docker, Proxmox, databases), available disk space, and OS/architecture.

## Communication

Agents connect outbound to the BackupOS web app over HTTPS. No inbound firewall rules required on agent machines.

## Updates

Agents check for updates and can be updated from the BackupOS UI without manual SSH. Self-update uses the same binary signing as the initial install.

# Restore specs

A restore spec is a YAML file that describes how to recover a system or service, step by step. Specs are stored in BackupOS and presented as a guided checklist in DR Mode during an incident.

## Why specs matter

Restore procedures documented in wikis rot. Specs live next to the job config, are version-controlled, and are tested during verification runs.

## Step types

| Type | Description |
|---|---|
| `restore_file` | Restore a file or directory from a snapshot |
| `restore_database` | Restore a database dump from a snapshot |
| `run_hook` | Execute a script on an agent |
| `notify` | Send a notification (webhook, email) |
| `manual` | A human-in-the-loop step with instructions |

## Failure modes

Each step can be configured with `on_failure: abort | continue | notify_only`. Critical steps abort the spec; non-critical steps continue and flag the failure for review.

# Verification

Verification is a scheduled test that restores data from a snapshot and checks the result. It's the only way to know a backup is restorable without doing it manually.

## How it works

1. BackupOS picks the most recent snapshot for the job
2. An agent restores one or more files to a sandbox target path
3. Optional validation hooks run (e.g. check a database can be opened)
4. Result: **passed** or **failed**

## Sandbox targets

Verification restores to a temporary path on the agent — not to production. After the test, the temporary files are deleted.

## Alert on failure

If verification fails, BackupOS fires a critical alert. Your backup succeeded but the restore did not work — you need to investigate.
```

# Health score

The health score is a 0–100 number (displayed as A–F) that summarises how healthy a backup setup is. It's computed per agent and per repository.

## Factors

| Factor | Weight | Description |
|---|---|---|
| Recent run success rate | 40% | Percentage of runs that succeeded in the last 30 days |
| Snapshot freshness | 30% | How recently the last successful snapshot was taken |
| Verification results | 20% | Pass rate of restore verification tests |
| Pre-flight pass rate | 10% | Percentage of pre-flights that passed |

## Letter grades

| Score | Grade |
|---|---|
| 90–100 | A |
| 80–89 | B |
| 70–79 | C |
| 60–69 | D |
| 0–59 | F |

A score below 70 triggers a warning badge. Below 50 triggers a critical alert if alert rules are configured.

# Tighten retention to save costs

Go to **Repositories → [repository] → Retention policy**. Configure the Restic forget policy:

| Setting | Example | Effect |
|---|---|---|
| Keep last N | 7 | Always keep the 7 most recent snapshots |
| Keep daily for | 30 days | Keep one snapshot per day for 30 days |
| Keep weekly for | 8 weeks | Keep one per week for 8 weeks |
| Keep monthly for | 12 months | Keep one per month for 12 months |
| Keep yearly | 3 | Keep one per year for 3 years |

BackupOS applies the policy on a schedule (default: daily). Pruning actually reclaims storage — without prune, forget only removes the index entry.

      fontSize: 14,
    }}>
      <MDXRemote source={source} />
    </article>
  )
