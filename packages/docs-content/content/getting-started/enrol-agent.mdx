# Enroll your first agent

The BackupOS agent runs as a system service on each host you want to back up. It connects outbound over WebSocket to the BackupOS server, picks up jobs, runs `restic` against the destination repository, and reports progress.

Linux agents are a Node.js bundle (`agent.js`) installed at `/opt/backupos-agent/`. Windows agents are a native binary (`backupos-agent.exe`) under `Program Files`. Both expose the same agent protocol.

## Linux

### 1. Generate an enrollment token

In the BackupOS UI, go to **Agents** and click **Enroll agent**.

Give the agent a name (typically the hostname, e.g. `web01` or `nas`), pick a default repository if you want one auto-attached, and click **Generate token & enroll**.

The next page shows the agent's detail view with a one-line install command, pre-filled with the new token.

### 2. Run the install command on the target host

Copy the command from the agent detail page and run it on the host you're enrolling. It looks like this:

```bash
curl -fsSL https://your-backupos-host:3093/install.sh | sudo BACKUPOS_TOKEN=<token> bash
```

The script:

1. Validates that `node` is on `PATH` (Node.js 20+ required) and installs it via your package manager if not.
2. Downloads the agent bundle to `/opt/backupos-agent/agent.js`.
3. Writes `/opt/backupos-agent/.env` with the server URL and token.
4. Creates the systemd unit `/etc/systemd/system/backupos-agent.service`.
5. Enables and starts the unit.

<Tip>The token is a one-time secret — once the agent has connected, it's exchanged for a long-lived credential held in `.env`. Treat the install command like an SSH key: paste it into the target host's terminal, don't share it.</Tip>

### 3. Verify

On the BackupOS UI, the **Agents** page should show the new agent with status **Connected** within a few seconds.

On the host:

```bash
systemctl is-active backupos-agent
# Should print: active

journalctl -u backupos-agent -f
# Watch for: "[agent] connected to <ws-url>"
```

### Updating a Linux agent

The install script is idempotent — re-running it upgrades in place. You can also run the script's update mode without re-piping from curl:

```bash
sudo bash /opt/backupos-agent/install.sh update
```

This re-fetches the latest installer from the server, replaces the agent bundle, and restarts the unit. Existing token and config are preserved.

## Windows

### 1. Generate an enrollment token

Same flow as Linux: **Agents → Enroll agent → Generate token & enroll**. The agent detail page shows both Linux and Windows install snippets.

### 2. Run the PowerShell installer

Copy the PowerShell snippet from the agent detail page and run it in an **elevated PowerShell** prompt on the Windows host. It looks like this:

```powershell
iex ((iwr "https://your-backupos-host:3093/install.ps1").Content) -EnrollmentToken "<token>"
```

The script downloads `backupos-agent-windows-x64.exe` to `C:\Program Files\BackupOS Agent\`, registers a Windows Service named **BackupOS Agent**, and starts it.

### 3. Verify

In **services.msc**, **BackupOS Agent** should be **Running**. The agent should also appear in the BackupOS UI as **Connected**.

## Capabilities and what each agent can back up

Once connected, click an agent in the UI to see its capabilities. The agent self-reports:

- Available source types (filesystem, Docker containers, Docker volumes, databases, hypervisor APIs)
- Free disk space on key mount points
- OS, architecture, kernel version
- App-aware hooks present on the host (e.g. `pg_dumpall` for PostgreSQL, `mysqldump` for MySQL)

This is what the **Job creation** form uses to decide which sources you can pick for a job running on this agent.

## Troubleshooting

**Agent appears but shows Disconnected** — the agent host can't reach the BackupOS server's WebSocket port. Confirm the host can reach `https://your-backupos-host:3093/` over HTTPS. If you're using a self-signed cert, see the next item.

**`curl: SSL certificate problem: self signed certificate`** — for testing only, add `-k` to the curl command. For production, put BackupOS behind a reverse proxy with a real TLS cert (Let's Encrypt, internal CA, etc.). The agent will refuse self-signed certs once connected unless you explicitly configure trust.

**Agent connects, then drops every few seconds** — usually a token mismatch. The agent's `.env` file might have an old token. Regenerate from the UI (delete the agent, run a fresh enrollment) or edit `/opt/backupos-agent/.env` and `systemctl restart backupos-agent`.

**`node: command not found` during install** — the install script tries to install Node automatically via `apt` / `dnf` / `yum`. If your distro isn't covered, install Node 20+ manually and re-run the script.

**Windows agent fails to start** — check the Windows Event Log under **Applications and Services Logs → BackupOS Agent** for the error. Typical causes are firewall blocking outbound 443/3093, or an antivirus quarantining `backupos-agent.exe`.
