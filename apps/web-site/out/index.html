Restic is great. Managing it isn't.

Four problems that bite every self-hoster eventually.

🔕
Silent failures
Cron jobs fail, no one notices. Data is gone when you need it most.
🗂
Repo sprawl
Dozens of Restic repos across machines with no central view of health or size.
⏱
Manual scheduling
Writing cron syntax and prune policies by hand for every new repo.
🔑
Key management
Repository passwords stored in plaintext scripts or forgotten entirely.

Everything Restic needs to be production-ready

Built for self-hosters who treat their data seriously.

🗓
Job scheduler
CRON expressions with UI preview. Run on-demand or on schedule.
📬
Email alerts
Configurable SMTP. Get notified on failure, success, or both.
🔍
Health checks
Automated `restic check` jobs with pass/fail history.
🔐
Password escrow
AES-256-GCM encrypted key storage with passphrase recovery.
✂️
Prune policies
Keep N daily / weekly / monthly snapshots, applied per job.
📊
Dedup stats
Per-repo size, raw size, and deduplication ratio at a glance.
🔄
Restore wizards
Guided file, database, and full-host restore with DR runbooks.
📋
Audit log
Immutable log of every backup run, check, and configuration change.

Up and running in 60 seconds

No external database required — SQLite included.

docker run -d \
  --name backupos \
  -p 3000:3000 \
  -v backupos-data:/data \
  ghcr.io/backupos/backupos:latest

Then open http://localhost:3000
